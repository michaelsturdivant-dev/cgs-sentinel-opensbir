# Wix Sales Funnel and Passive Income Plan

## Commercial Architecture

GitHub is the public proof layer.

Wix is the buyer-facing front end.

CGS Sentinel OpenSBIR™ is the standalone protected operating engine and reference model behind the offer.

The public visitor should see a clear problem, a clear offer, a clear transformation, and a clear way to buy or request a readiness sprint. The visitor should not see private prompts, advanced routing rules, customer workflows, sensitive data, or commercial playbooks.

## Front-End Promise

The Wix page should provide X:

- A clear explanation of the OpenSBIR model
- A readiness assessment
- A paid service offer
- A link to the GitHub proof layer
- A buyer intake form
- A scheduling link
- A payment or deposit option
- A service menu
- A trust and data-rights explanation

## Back-End Reality

The protected back end is Sentinel itself. It should be presented as one standalone CGS operating engine with modular capabilities, not as a system linked to other proprietary engines.

Sentinel can support public-safe modules for:

- SBIR readiness
- Grant operations
- OTA concept packaging
- Compliance routing
- Work-item and POAM tracking
- Artifact planning
- Strategic planning committee support
- Government contracting support
- Community infrastructure planning

## Passive Income Ladder

Start with high-touch consulting. Convert repeated work into semi-passive products.

1. Free GitHub reference model
2. Low-cost readiness assessment
3. Paid template/toolkit bundle
4. 30-day readiness sprint
5. Monthly managed Sentinel support
6. Enterprise deployment

## Conversion Logic

The Wix page should not try to explain every technical detail. It should convert visitors by showing:

- The problem is real
- The structure is credible
- The GitHub model proves it is more than an idea
- The public layer is safe
- The private Sentinel capability is protected
- The buyer can start with a readiness assessment or sprint
